# MEMORY

## 1. 工作区定位
本工作区用于：
- 系统架构设计
- 工业AI应用开发
- 数据/模型/控制一体化方案设计
- 输出可落地的文档、代码骨架、接口契约、部署方案与运维规范

## 2. 长期工作原则
- 先明确业务边界、系统边界、数据边界，再做模块设计。
- 优先做 MVP，先打通主链路，再补非核心能力。
- 工业场景优先级：安全 > 稳定 > 可解释 > 可维护 > 性能优化。
- 涉及控制闭环时，默认从“建议模式”起步，再逐步走向“半闭环”，最后才考虑“全闭环”。
- 所有方案必须包含异常流、降级流、回退策略。
- 不接受只有模型、没有工程闭环的“伪工业AI方案”。

## 3. 默认架构输出要求
每次做架构设计，优先给出：
- 上下文图
- 容器图
- 组件图
- 核心时序图
- 数据流 / 控制流 / 异常流
- 接口清单
- 数据表或对象模型
- 部署拓扑
- 风险与边界

## 4. 工业AI设计基线
工业AI方案通常包含以下层次：
- 设备与采集层：PLC、传感器、仪表、边缘网关
- 数据接入层：OPC UA / MQTT / HTTP / 文件 / 数据库
- 数据治理层：点位映射、单位统一、缺失处理、异常检测、标签管理
- 特征与规则层：规则引擎、机理特征、统计特征、特征服务
- 模型服务层：训练、评估、部署、推理、版本管理
- 应用层：告警、预测、优化建议、报表、看板、控制建议
- 运维层：日志、监控、审计、权限、回滚、配置管理

## 5. 工业AI硬约束
- 必须考虑数据质量：缺失、噪声、时间对齐、漂移、单位不一致。
- 必须考虑模型可解释性与可信度输出。
- 必须考虑线上/线下口径一致性。
- 必须考虑模型失效时的规则回退。
- 必须考虑权限边界：建议、审批、下发、执行、审计分离。
- 必须考虑与现有 MES / ERP / LIMS / SCADA 的集成成本。

## 6. 长期偏好的交付风格
- 偏好结构化输出，少空话，多可执行内容。
- 偏好先给框架，再给细化内容。
- 偏好文档与代码骨架一起产出。
- 偏好明确列出假设、输入、输出、依赖、风险。
- 偏好中文输出，但代码、接口名、表名可保持英文。

## 7. 长期技术偏好（示例，可按你的栈改）
- 后端优先：FastAPI / Python
- 前端优先：Vue 或 React
- 数据库优先：PostgreSQL / 时序库 / Redis
- 部署优先：Docker Compose，后续可演进到 K8s
- 接口优先：REST 为主，必要时引入消息总线
- 文档优先：Markdown + Mermaid + OpenAPI + SQL DDL

## 8. 长期关注的系统风险
- 架构文档与代码实现脱节
- 数据字典与真实点位脱节
- 训练特征与线上推理特征不一致
- 模型版本、阈值、规则库版本混乱
- 自动控制越权
- 缺少日志、审计和回放机制

## 9. 长期未决事项
- 是否引入消息总线
- 是否引入时序数据库
- 模型服务与业务服务是否分离部署
- 在线推理与离线训练是否采用统一特征定义
- 是否引入数字孪生/机理模型作为二级校核

## 10. 不应写入本文件的内容
- 临时调试日志
- 当天会议流水
- 短期待办
- 大段报错原文
- 密码、Token、私钥、敏感配置
